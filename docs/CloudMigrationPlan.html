vms()

Returns a MigrationPlanActionsByEntityList containing only VM MigrationPlanEntitys which are in an ACTIVE state. The MigrationPlanEntity#allocation_actions and MigrationPlanEntity#consumption_actions will be filtered only to the MOVE actions. In practice, this should mean that this is a complete list of all ACTIVE state. For each VM, there will be 0-1 MOVE actions for allocation and 0-1 MOVE actions for consumption (depending on whether or not the VM was placed). These two values actions are used to show the "allocation" and "consumption" states of the VM in the vm-to-template-mapping output.
Source:

MigrationPlanActionsByEntityList

wait()

Waits for the market created by Plan#run to reach the "SUCCEEDED" state. This *could* result in an infinite loop, and doesn't have any provisions for a timeout. It also rapid-fire spams the API with no exponential backoff between checks. In short, it works, but it is heavy handed, and ripe for refactoring.

CloudMigrationPlan(from, to, name, optionsopt)

new CloudMigrationPlan(from, to, name, optionsopt)

Initializes a new CloudMigrationPlan.
Parameters:
Name Type Attributes Description
from object A group of VMs to be migrated to the cloud. Only required property on the object is 'uuid'
to object A group of PMs where the VMs should be migrated. Only required property on the object is 'uuid'
name string The name of the plan to be created
options object <optional>
 An object having 0 or more of the following options
  • exclude {object} A group to exclude. By default a group of VMs belonging to the 'to' group is created, and excluded.
  • byol {bool} A boolean indicating if BYOL should be used (Default: false)
  • hours_per_month {number} The number of hours per month (Default: 730)
Source:
  • plans.js, line 280
Example
var from = client.getSearchResults({q: "vms_vcenter"})[0];
var to = client.getSearchResults({q: "pms_azure-East US"})[0];
var plan = new CloudMigrationPlan(from, to, "DeleteMe", {byol: true});
